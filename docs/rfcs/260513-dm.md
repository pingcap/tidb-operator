## Summary

[DM (Data Migration)](https://docs.pingcap.com/tidb/stable/dm-overview) is PingCAP's data migration tool for migrating data from upstream MySQL/MariaDB databases to TiDB. This RFC proposes adding DM cluster management support to tidb-operator, covering the full lifecycle of dm-master and dm-worker processes.

Scope is **infrastructure only**: deploy and manage dm-master and dm-worker pods, PVCs, ConfigMaps, and Services. Task and source management remains out of scope and is handled externally via DM's OpenAPI.

## Motivation

Users who run TiDB clusters managed by tidb-operator also want to run DM clusters in the same Kubernetes environment under the same operational model (GitOps, kubectl, Helm). Currently they must manage DM deployments separately, with no operator support for rolling updates, TLS rotation, or lifecycle management.

### Goals

- Deploy and manage dm-master (forms an embedded etcd cluster) via new CRDs
- Deploy and manage dm-worker (connects to dm-master, stores relay logs on PVC) via new CRDs
- Support full lifecycle: creation, rolling update, deletion
- Support TLS (cluster-internal mTLS), matching existing component patterns
- Follow the existing Group+Instance controller pattern (same as TiCDC, TiKV, etc.)

### Non-Goals

- DM task management (create/pause/resume migration tasks) — use DM's OpenAPI directly
- DM source management (registering upstream MySQL sources) — use DM's OpenAPI directly
- Upstream/downstream TLS certificate provisioning — users mount cert Secrets via `overlay`
- Cross-cluster DM scheduling or multi-tenancy

## Proposal

### User Stories

#### Story 1

As a TiDB user running tidb-operator, I want to deploy a DM cluster alongside my TiDB cluster using the same kubectl/Helm workflow, so that I can migrate data from MySQL to TiDB without managing DM infrastructure separately.

#### Story 2

As a platform engineer, I want to perform a rolling update of dm-worker pods when upgrading the DM version, without disrupting ongoing migration tasks, so that I can maintain SLAs during upgrades.

### Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| dm-master bootstrap: all peer URLs must be known before the first pod starts | Follow PD pattern — compute `initial-cluster` from expected pod DNS names; use `join` for pods added to an existing cluster (detected via annotation `anno.core.pingcap.com/initial-cluster-num`) |
| Surge is unsafe for dm-worker: a new worker name breaks the `lastBound` history in dm-master etcd | Always use `maxSurge=0, maxUnavailable=1` for both DMGroup and DMWorkerGroup rolling updates |
| DMWorker needs to know dm-master address at config generation time | `DMWorkerGroup.spec.dmGroupRef` points to the DMGroup by name; controller resolves the headless Service DNS from it |
| TLS cert Secret naming must match existing conventions | Reuse `coreutil.IsTLSClusterEnabled` and `apicall.GetClientTLSConfig` — no new TLS logic |

## Design Details

### API

Four new CRDs are introduced, following the Group+Instance pattern used by all other components:

**dm-master** (`DMGroup` / `DM`):

```yaml
apiVersion: core.pingcap.com/v1alpha1
kind: DMGroup
metadata:
  name: dm-master
spec:
  cluster:
    name: basic
  replicas: 3
  template:
    spec:
      version: v8.5.2
      resources:
        cpu: "500m"
        memory: 1Gi
      dataVolume:         # persistent volume for embedded etcd data (required)
        name: data
        storage: 10Gi
      config: |           # user TOML merged after operator-managed fields
        log-level = "info"
```

**dm-worker** (`DMWorkerGroup` / `DMWorker`):

```yaml
apiVersion: core.pingcap.com/v1alpha1
kind: DMWorkerGroup
metadata:
  name: dm-worker
spec:
  cluster:
    name: basic
  dmGroupRef:             # references the DMGroup to join (required)
    name: dm-master
  replicas: 2
  template:
    spec:
      version: v8.5.2
      resources:
        cpu: "500m"
        memory: 1Gi
      relayVolume:        # persistent volume for relay logs (required)
        name: relay
        storage: 50Gi
```

**Ports**:

| Component | Port | Name | Purpose |
|-----------|------|------|---------|
| dm-master | 8261 | `dm-master` | OpenAPI / gRPC client access |
| dm-master | 8291 | `dm-master-peer` | Embedded etcd peer communication |
| dm-worker | 8262 | `dm-worker` | gRPC + HTTP (cmux) |

**Services** created by the DMGroup controller:
- Headless Service `<dmgroup-name>-peer` (port 8291) — etcd peer discovery
- Client Service `<dmgroup-name>` (port 8261) — worker join address and user access

**Probes**:
- dm-master: `GET /api/v1/cluster/info` on port 8261 (the OpenAPI endpoint; `/health` on 8291 is the embedded etcd's endpoint, not dm-master's)
- dm-worker: `GET /status` on port 8262 (returns plain-text version info via the HTTP mux; port 8262 uses cmux to multiplex gRPC and HTTP)

**Image**: `pingcap/dm:${version}` for both dm-master and dm-worker.

### dm-master Bootstrap

dm-master uses an embedded etcd cluster that requires `initial-cluster` on first start and `join` for members added later — the same challenge as PD.

**Bootstrap detection**: An annotation `anno.core.pingcap.com/initial-cluster-num` is set on the DMGroup by the group controller when first creating instances. If this annotation is present on the DM instance, the ConfigMap is generated with `initial-cluster`; otherwise `join` pointing to the client Service DNS is used.

**`initial-cluster` value** (computed from expected pod DNS names):
```
<pod-0>=http://<pod-0>.<headless-svc>.<ns>.svc:8291,<pod-1>=http://...
```

**`join` value** (for all subsequent pods):
```
<dmgroup-name>.<ns>:8261
```

### Configuration Management

The operator manages a set of fields that are always computed from the cluster topology and must not be overridden by users:

**dm-master managed fields** (always set by operator):
- `name`, `master-addr`, `advertise-addr`
- `peer-urls`, `advertise-peer-urls`
- `data-dir`, `initial-cluster` / `join`
- `ssl-ca`, `ssl-cert`, `ssl-key` (when TLS enabled)
- `openapi` (always `true`)

**dm-worker managed fields**:
- `name`, `worker-addr`, `advertise-addr`
- `join` (dm-master client Service address)
- `relay-dir`
- `ssl-ca`, `ssl-cert`, `ssl-key` (when TLS enabled)

User-provided `spec.config` (TOML blob) is decoded, validated for conflicts with managed fields, and merged — managed fields take precedence. If the user attempts to set a managed field, the ConfigMap task returns `SFail`.

Config packages:
- `pkg/configs/dm/` — dm-master config overlay
- `pkg/configs/dmworker/` — dm-worker config overlay

### Rolling Update Strategy

Both DMGroup and DMWorkerGroup use a fixed strategy: **`maxSurge=0, maxUnavailable=1`**.

Surge is explicitly unsafe for dm-worker: dm-master tracks each worker by its advertised name in embedded etcd. A new worker pod with a different name would appear as a new registration, breaking the `lastBound` source-to-worker mapping that enables transparent task resumption after pod restart.

dm-master tolerates one unavailable member during rolling updates because its embedded etcd cluster maintains quorum with `floor(N/2)` members available.

No `UpgradeChecker` (PD readiness gate) is needed — dm-master has no PD dependency.

### TLS Handling

Cluster-internal TLS (mTLS between dm-master and dm-worker, and between the operator and DM's API) follows the existing pattern:
- Controlled by `Cluster.spec.tlsCluster.enabled`
- TLS config obtained via `apicall.GetClientTLSConfig(ctx, c, cluster)`
- Cert paths mounted at `/var/lib/dm-tls/` in both dm-master and dm-worker pods

Upstream/downstream TLS (connecting to source MySQL or target TiDB) is a task-level concern:
- **Option A** (zero operator involvement): Use base64-inline fields (`ssl-ca-base64` etc.) in source/task config submitted via DM OpenAPI
- **Option B**: Mount cert Secrets into dm-worker pods via `DMWorkerGroupTemplateSpec.overlay`, then reference the mounted paths in source/task config

No new CRD fields are needed for either option — the existing `overlay` field (shared by all components) handles pod spec customization.

### Test Plan

**Unit tests** (per controller `tasks/` package, mirroring TiCDC):
- `cm_test.go` — ConfigMap generation: managed fields injected, user fields merged, managed field conflict rejected
- `svc_test.go` — Service creation: headless peer service and client service
- `boot_test.go` — Bootstrap annotation detection: `initial-cluster` vs `join` selection
- `updater_test.go` — Rolling update: `maxSurge=0` respected, scale out/in

**Config tests** (`pkg/configs/dm/`, `pkg/configs/dmworker/`):
- Overlay correctness: all managed fields present, TLS paths injected when enabled, user config merged

**CRD validation tests** (`tests/validation/`):
- `dmgroup_test.go` — cluster reference required, name length limit, minReadySeconds
- `dmworker_test.go` — `spec.dmGroupRef.name` required, cluster reference required

**e2e tests** (`tests/e2e/dm/`, label `c:DM`):
- Basic lifecycle: apply DMGroup + DMWorkerGroup → pods reach Running; verify dm-worker registers with dm-master via OpenAPI
- Delete cleanup: delete DMWorkerGroup → pods and PVCs removed; delete DMGroup → pods and ConfigMaps removed
- Scale out/in: increase/decrease `replicas`, verify pod count converges
- Rolling update: version bump triggers pod recreation one-by-one without simultaneous unavailability

### Feature Gate

No feature gate — DM support is enabled by deploying DM CRDs and the operator handles them when present.

## Drawbacks

DM is a separate product from TiDB and has its own release cadence. The operator must track DM's config schema changes. However, since users provide raw TOML for user-facing config and the operator only manages a fixed set of structural fields, the coupling is limited.

## Alternatives

**Helm chart only**: Deploy DM via a standalone Helm chart without operator support. This loses rolling update, TLS rotation, and lifecycle integration with the TiDB cluster CR. Rejected because operator users expect consistent management across all PingCAP components.

**Single CRD for dm-master + dm-worker**: Combine both into a `DMCluster` CRD. Rejected because the Group+Instance pattern requires separate CRDs per process type, and mixing two distinct processes with different volume, port, and bootstrap requirements into one CRD increases complexity without benefit.

**Reuse TiCDCGroup for dm-master**: dm-master is structurally similar to TiCDC (stateless-ish, single port, headless service). Rejected because dm-master has a second peer port (8291), requires a data PVC for embedded etcd, and has distinct bootstrap semantics (`initial-cluster`) that would require too many special cases in a shared controller.
